## AISH とは何か

AISH は、LLM によって Linux の CUI 体験を強化する **CUI 自動化フレームワーク**です。  
Rust 製の `ai` / `aish` バイナリとして提供され、ターミナルで行う日常的な開発作業を、自然言語経由で補助・自動化することを目的としています。

- **`aish`**: 「ターミナルの状態と履歴を記録するシェル」。  
  - 各コマンドの標準入出力を **セッション** として保存します。
  - セッションごとにディレクトリ（XDG 時は `~/.local/state/aish/session/<id>/`、AISH_HOME 時は `$AISH_HOME/state/session/<id>/`）が作られ、ログやメタ情報が蓄積されます。
- **`ai`**: 「現在のコンテキストを踏まえて LLM に問い合わせるコマンド」。  
  - `aish` で蓄積されたセッション情報をもとに、LLM への **1 回の問い合わせ** を行います。
  - 単発の質問だけでなく、コードレビューやログ解析などの複雑なタスクも扱えます。

この 2 つを組み合わせることで、

- 端末で起きていることをそのまま LLM に共有しつつ質問する
- 過去のコマンド・出力を文脈として踏まえた提案やレビューを受ける

といった「**コンテキストを保ったまま対話できる開発体験**」を実現します。

## 想定ユースケース

- **エラー調査・バグ調査**
  - `aish` 上でテストやコマンドを実行し、そのログを踏まえて `ai` に原因の推測や次の一手を相談する。
- **コードレビュー・改善提案**
  - 変更点やファイルの内容を踏まえたレビューコメントを `ai` に生成させる。
- **ドキュメントやコメントの生成**
  - 既存コードやターミナルログを参照しながら、設計メモ・リリースノート・コメント案を生成させる。
- **対話的なリファクタリング支援**
  - 「この関数をこう変えたい」といった自然言語の相談から、具体的な変更案やチェックポイントを出してもらう。

## `aish` と `ai` の関係（セッションと LLM リクエスト）

高レベルには、次のような関係になっています。

1. ユーザーは `aish` を起動し、通常のシェルと同じようにコマンドを実行する。
2. `aish` は、各コマンドの入出力をセッションディレクトリに記録する。
3. ユーザーが同じターミナルから `ai` を実行すると、`AISH_SESSION` に紐づくセッションが検出される。
4. `ai` は、そのセッションの履歴や設定をもとに LLM へ問い合わせ、結果を標準出力に表示する。

このため、`ai` を単独で呼び出すこともできますが、**`aish` 上で作業し、その流れの中で `ai` を呼ぶ**のが、AISH らしい使い方になります。

## はじめて使うときの読み進め方

1. **この `overview.md`**  
   - AISH 全体のコンセプトと `aish` / `ai` の関係をつかみます。
2. **`commands.md`**  
   - どのコマンドで何ができるか、ざっくりとした地図を把握します。
3. **`aish-usage.md`**  
   - セッションの概念と、日常的なターミナル作業をどう `aish` に載せ替えるかを確認します。
4. **`ai-usage.md`**  
   - `ai` でどのような質問・依頼をすれば、どんな応答が返ってくるかを、具体例とともに学びます。

この順番で読むことで、「AISH とは何か」から「日々の開発作業にどう組み込むか」までを段階的に理解できるようになっています。
